# AVL树与B树概述

## 为什么需要平衡树？

| 数据结构 | 优势 | 劣势 |
|----------|------|------|
| 数组 | 查询O(1) | 插入删除O(n) |
| 链表 | 插入删除O(1) | 查询O(n) |
| BST | 查询插入O(logn) | 有序数据退化为O(n)链表 |

**平衡树**就是为了平衡静态操作与动态操作的差距。

---

## 一、二叉查找树（BST）

### 定义条件

- 左子树值均小于根节点
- 右子树值均大于根节点
- 左右子树也是BST
- 无键值相等的节点

### 问题

**最坏情况**：插入有序数据退化为线性链表，查询效率O(n)

---

## 二、AVL树（平衡二叉树）

### 定义

带有平衡条件的BST，通过**平衡因子**判断是否平衡：

```
平衡条件：所有节点的左右子树高度差不超过1
```

### 特性

| 特性 | 数值 |
|------|------|
| 查找效率 | O(log₂n) |
| 插入旋转 | O(logn)次 |
| 删除旋转 | O(logn)次（最多） |
| 100万节点高度 | 28层 |

### 局限性

- 维护严格平衡代价大
- 旋转非常耗时
- **适用场景**：插入删除少，查找多

### 应用

- Windows进程地址空间管理

---

## 三、红黑树

参见 [[summaries/red-black-tree-detailed]]

### 核心差异

| 维度 | AVL树 | 红黑树 |
|------|-------|--------|
| 平衡严格度 | 严格（差≤1） | 弱平衡（≤2倍） |
| 旋转次数 | 多（logn级） | 少（≤1次） |
| 适用场景 | 查找多 | 增删查都频繁 |

### 红黑树应用

- C++ STL map/set
- Linux CFS调度器（进程控制块）
- Linux虚拟内存区域管理
- epoll（sockfd管理）
- Nginx timer管理
- Java TreeMap

---

## 四、B树（多叉树）

### B-Tree定义（m阶）

```
1. 非叶子节点最多M个儿子（M>2）
2. 根节点儿子数[2, M]
3. 非根非叶子节点儿子数[M/2, M]
4. 每节点存放[M/2-1, M-1]个关键字
5. 非叶子节点关键字数=儿子指针数-1
6. K[1] < K[2] < ... < K[M-1]
7. 所有叶子节点位于同一层
```

### B-Tree特性

- 关键字分布在整个树中
- 搜索可能在非叶子节点结束
- 性能等价于全局二分查找

### B+Tree定义

B+树是B树的变体：

```
1. 非叶子节点子树指针数=关键字数
2. 所有关键字都在叶子节点出现
3. 叶子节点增加链指针
4. 非叶子节点只是索引层
```

### B+Tree特性

| 特性 | 说明 |
|------|------|
| 稠密索引 | 所有关键字在叶子节点链表中 |
| 稀疏索引 | 非叶子节点仅作索引 |
| 范围查询 | 高效遍历叶子节点链表 |

---

## B树 vs B+树对比

| 维度 | B树 | B+树 |
|------|-----|------|
| 关键字数 | m-1 | m |
| 数据位置 | 所有节点 | 仅叶子节点 |
| 分支节点 | 存关键字信息 | 仅存索引+指针 |
| 查找结束 | 可能非叶子 | 必须到叶子 |
| 范围查询 | 效率低 | **高效遍历叶子链表** |

---

## 为什么B+树更适合文件/数据库索引？

### 1. 磁盘读写代价更低

- B+树内部节点无数据指针，更小
- 同一盘块容纳更多关键字
- IO读写次数更少

### 2. 查询效率更稳定

- 所有查询都走根到叶子的完整路径
- 每个数据查询效率相同

### 3. 范围查询效率高

- 只需遍历叶子节点链表
- B树不支持高效范围查询

## 应用场景总结

| 树结构 | 典型应用 |
|--------|----------|
| AVL树 | Windows进程地址空间 |
| 红黑树 | Linux调度器/epoll、STL map、Java TreeMap |
| B树 | 文件系统索引（部分） |
| B+树 | **数据库索引**、MySQL InnoDB |
