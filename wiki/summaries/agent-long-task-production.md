# Agent 期任务生产失败分析与解决

> ⚠️ **低可信度警告**：本文来源于 AI 自主收集的网络博客，未经用户审核确认。请谨慎参考，建议阅读原文验证。

超过 30 秒的 Agent 任务面临完全不同的系统问题。大多数团队在发布产品后才撞上这堵墙。

---

## 一、复合概率的残酷数学

### 步骤成功率 vs 最终成功率

| 步骤数 | 每步 95% 成功 | 每步 99% 成功 |
|--------|--------------|--------------|
| 10 步 | 60% | 90% |
| 20 步 | 36% | 82% |
| 30 步 | 21% | 74% |

**核心结论**：问题不是模型质量，而是复合概率对顺序系统的惩罚。

**解决思路**：
- 更短、有界工作流 > 开放式规划
- 幂等单步重试 > 端到端重新运行
- 阶段边界 Checkpoint > 重启所有工作

**研究数据**：68% 生产从业者将 Agent 限制在有界工作流，而非开放式规划。

---

## 二、30 秒同步边界

### 同步 HTTP 适用场景

- ✓ 30 秒内完成的 AI 任务
- ✓ 客户端阻塞 → Agent 运行 → 响应返回
- ✓ 简单、可靠

### 超过边界后的崩溃点

| 问题 | 说明 |
|------|------|
| 负载均衡器超时 | 默认配置通常 30-60 秒 |
| 移动客户端断开 | 用户关闭应用或网络切换 |
| Serverless 执行限制 | AWS Lambda 15 分钟上限 |
| 用户刷新页面 | 触发第二次请求，重复执行 |

### 正确模式

```
同步请求 → 立即返回 Task ID
            ↓
Agent 在后台 Worker 运行（与 HTTP 请求断开）
            ↓
客户端轮询 / Webhook 获取完成状态
```

### AI Agent 特殊管理需求

| 需求 | 说明 |
|------|------|
| **LLM 上下文序列化** | 不能保存在进程内存中 |
| **工具调用结果持久化** | 重试时不重新执行副作用 |
| **进度信号送达** | 前端已与请求断开 |
| **双重速率限制** | RPM + TPM，队列计算"何时可安全运行" |

## 三、Checkpointer vs 持久化执行

### 核心区别

| 维度 | Checkpointer | 持久化执行 |
|------|--------------|-----------|
| **承诺** | "保存了状态，接下来交给你" | "工作流一定运行到完成" |
| **故障恢复** | 需人或机制检测、手动触发 | 运行时自动心跳检测、回放恢复 |
| **重复执行** | 多 Worker 可能重复执行 | 幂等保证，不会重复 |
| **编排责任** | 你提供编排逻辑 | 系统为你处理 |

### 框架对比

| 框架 | 类型 | 说明 |
|------|------|------|
| LangGraph PostgresSaver | Checkpointer | 概念强大，实践中需自己编排 |
| Temporal | 持久化执行 | 自动故障检测、回放恢复 |
| Trigger.dev | 持久化执行 | 更贴合 AI 场景的人机工程学 |
| Inngest | 持久化执行 | 新兴专用平台 |

### LangGraph 团队迁移案例

最初在 LangGraph + Redis 构建 Agent 系统 → "概念强大但实践脆弱" → 迁移到 Temporal 消除自定义重试恢复逻辑。

## 四、幂等性是首要关注

### 非幂等副作用的隐蔽故障

```
Agent 调用支付 API → Worker 崩溃（未更新 Checkpoint）
    ↓
重试 → 再次调用支付 API → 重复扣款！
```

### 解决模式

**每次外部写入携带幂等键**：
- 键绑定：工作流 ID + 步骤编号
- 同键二次调用 → 返回原始结果的空操作（No-op）

### 实践清单

- ✓ 每次外部写入需要幂等键
- ✓ 每个队列入队时去重
- ✓ 每次重试从持久化状态提取上下文，不从头重建

## 五、人机协作（Human-in-the-Loop）

### 系统设计挑战

| 问题 | 说明 |
|------|------|
| Worker 必须释放上下文 | 不能占用 Worker 等待人工 |
| 存储足够恢复状态 | 干净恢复，不丢失进度 |
| 无限期等待 | 可能几小时或几天 |

**核心**："等待人工"是执行模型的一等状态，不是轮询数据库的变通方法。

### 框架支持

| 框架 | 机制 |
|------|------|
| LangGraph | `interrupt()` 函数在定义节点暂停，输入到达时恢复 |
| Temporal | Signal 处理器，工作流可无限期阻塞 |

### 中断点设计

| 中断点 | 触发条件 |
|--------|----------|
| 置信度阈值异常 | Agent 不确定意图，先询问 |
| 不可逆操作边界 | 删除、修改生产系统、触发支付 |
| 成本检查点 | 大量 Token 消耗前验证进度 |
| 模糊输入 | 暴露模糊性而非幻想解决方案 |

## 六、35 分钟衰减问题

### 经验性可靠性悬崖

AI Agent 连续运行约 35 分钟后，可靠性下降。

**假设原因**：
- 上下文窗口饱和
- 累积推理漂移（每个决策包含之前所有决策噪声）

### 解决方案：有边界会话

| 模式 | 说明 |
|------|------|
| 初始化 Agent | 第一个会话建立基础状态 |
| 编码 Agent | 后续会话从 git 提交历史 + 进度文档重建上下文 |
| 增量进展 | 每个会话有边界，外部状态重建 |

**核心原则**：
- 运行 2 小时的 Agent ≤ 两个各 1 小时 + 干净移交的 Agent
- 有边界会话更可控、更易调试

## 七、生产部署清单

### 必须优先考虑

| 项目 | 说明 |
|------|------|
| 异步设计 | 超过 30 秒必须异步，立即返回 Task ID |
| 状态持久化 | 恢复所需的所有状态都要持久化 |
| 幂等写入 | 每次外部写入携带幂等键 |
| 中断点定义 | 提前设计到工作流拓扑中 |
| 会话长度限制 | 分解为有边界会话 + 干净移交 |

### 不是事后优化

**这些都是基石**，让其他一切变得可靠。事后修补比一开始正确构建更昂贵——同步假设被硬编码在 API 响应契约、前端状态管理、错误处理、成本归属中。
