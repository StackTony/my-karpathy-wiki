# 红黑树详解

## 简介

红黑树是一种**自平衡的二叉查找树**，由Rudolf Bayer于1978年发明。它在O(logN)时间内完成查找、增加、删除等操作，是业界应用最广泛的平衡树结构。

### 为什么需要红黑树？

| 问题 | 解决方案 |
|------|----------|
| BST插入有序数据退化为链表（O(N)） | 红黑树保证最长路径≤2倍最短路径 |
| AVL树严格平衡导致频繁旋转 | 红黑树弱平衡，旋转次数更少 |

## 五大性质（核心）

| 性质 | 内容 |
|------|------|
| 1 | 节点是红色或黑色 |
| 2 | 根节点是黑色 |
| 3 | 叶子节点（空节点）是黑色 |
| 4 | 红色节点的子节点都是黑色（无连续红节点） |
| 5 | 从任一节点到叶子的所有路径包含相同数目黑色节点 ⭐ |

**关键结论**：最长路径不超过最短路径的2倍，因而近似平衡。

## 与4阶B树的等价性

红黑树可等价变换为4阶B树（2-3-4树）：

```
红黑树节点          →    B树节点
黑色节点 + 红色子节点 →    合并为1个B树节点
黑色节点在中间，红色在两边
```

这种等价性是理解红黑树操作的关键视角。

## 旋转操作

### 左旋
将节点旋转为其右孩子的左孩子：
```
1. M的左孩子指向E的右孩子
2. E的右孩子指向M
```

### 右旋
将节点旋转为其左孩子的右孩子：
```
1. M的右孩子指向E的左孩子
2. E的左孩子指向M
```

## 插入操作详解

### 新节点颜色
- **插入节点默认红色**（只可能违反性质4，便于修复）
- 根节点插入为黑色

### 12种插入情况分类

| 类别 | 条件 | 处理 |
|------|------|------|
| 4种满足性质 | parent为黑色 | 无需处理 |
| 4种非上溢 | parent红，uncle非红 | 染色+旋转 |
| 4种上溢 | parent红，uncle红 | 染色+grand向上合并 |

### 非上溢情况修复

| 情况 | 修复步骤 |
|------|----------|
| LL（父左，新左） | parent染黑，grand染红，grand右旋 |
| RR（父右，新右） | parent染黑，grand染红，grand左旋 |
| LR（父左，新右） | 新节点染黑，grand染红，parent左旋+grand右旋 |
| RL（父右，新左） | 新节点染黑，grand染红，parent右旋+grand左旋 |

### 上溢情况修复

```
1. parent、uncle染成黑色
2. grand向上合并（染红），当做新插入节点继续处理
3. 若上溢到根，根染黑即可
```

## 删除操作详解

删除节点一定在最后一层（B树特性）。

### 删除情况分类

| 情况 | 处理 |
|------|------|
| 删除红色节点 | 直接删除，无需调整 |
| 删除有1个红色子节点的黑色节点 | 用红色子节点替代，染黑 |
| 删除黑色叶子节点（根节点） | 直接删除 |
| 删除黑色叶子节点（兄弟黑色） | 见下表 |
| 删除黑色叶子节点（兄弟红色） | 先转换兄弟为黑色情况 |

### 兄弟节点为黑色的修复

| 子情况 | 条件 | 修复 |
|--------|------|------|
| 借节点 | 兄弟有红色子节点 | 旋转，中心节点继承父颜色，左右染黑 |
| 父节点下溢合并 | 兄弟无红色子节点，父红色 | 兄弟染红，父染黑 |
| 父节点下溢合并 | 兄弟无红色子节点，父黑色 | 父节点当做被删除节点继续处理 |

## AVL树 vs 红黑树对比

| 特性 | AVL树 | 红黑树 |
|------|-------|--------|
| 平衡标准 | 严格：高度差≤1 | 弱平衡：最长≤2倍最短 |
| 查找效率 | 更高（树更矮） | 较高 |
| 插入删除旋转 | O(logn)次旋转 | **O(1)次旋转** |
| 适用场景 | 查找多，插入删除少 | 查找、插入、删除都频繁 |

### 100万节点的最大高度

- AVL树：28层
- 红黑树：40层

## 应用场景

| 系统/框架 | 应用 |
|-----------|------|
| **Linux CFS调度器** | 用红黑树管理进程控制块，按vruntime排序 |
| **Linux epoll** | 用红黑树管理sockfd，支持快速增删改查 |
| **Linux虚拟内存** | 进程虚拟内存区域存储在红黑树 |
| **Nginx** | 用红黑树管理timer |
| **Java TreeMap** | 底层实现 |
| **C++ STL map/set** | 底层实现 |

## 时间复杂度

| 操作 | 复杂度 | 旋转次数 |
|------|--------|----------|
| 查找 | O(logn) | 0 |
| 插入 | O(logn) | ≤1 |
| 删除 | O(logn) | ≤1 |

## 与Linux内核的关联

- [[concepts/cfs-scheduler]] 使用红黑树组织进程，vruntime作为key
- [[summaries/linux-scheduler]] 调度器的核心数据结构
- 红黑树的选择原因：进程调度需要频繁插入/删除，红黑树O(1)旋转更优
